TCP vs UDP

📖 Explicação

TCP (Transmission Control Protocol) e UDP (User Datagram Protocol) são os dois protocolos da camada de Transporte. Eles operam sobre o IP e entregam dados entre aplicações. A diferença fundamental está na confiabilidade vs. velocidade.

CaracterísticaTCPUDP
ConexãoOrientado a conexão (three-way handshake)Sem conexão (connectionless)
ConfiabilidadeGarante entrega e ordem corretaMelhor esforço, sem garantias
Confirmação (ACK)Sim — retransmite se não receber ACKNão

Uso típicoHTTP/S, SSH, FTP, e-mail, banco de dadosDNS, DHCP, streaming, VoIP, jogos
Cabeçalho20–60 bytes8 bytes
💡 Analogia

TCP é como uma ligação telefônica: você estabelece a chamada, fala, confirma que o outro entendeu, e encerra formalmente. UDP é como gritar em um megafone na rua: você transmite sem saber se alguém ouviu, mas é muito rápido.

⚡ Regra de ouro

Use TCP quando perder dados é inaceitável (transferência de arquivos, autenticação, transações). Use UDP quando velocidade e baixa latência importam mais que perfeição (streaming ao vivo, chamadas de voz, jogos online).

DNS — Domain Name System

📖 Explicação

O DNS é o sistema que traduz nomes de domínio legíveis por humanos (google.com) em endereços IP (142.250.78.14) que os computadores usam. Sem DNS, você precisaria memorizar IPs numéricos para cada site.

Como funciona uma consulta DNS:

  1. Você digita google.com
  2. O sistema operacional verifica o cache DNS local — já consultou antes?
  3. Se não, consulta o servidor DNS recursivo do provedor (ou 8.8.8.8, 1.1.1.1)
  4. O recursivo consulta um servidor raiz (root) — quem gerencia .com?
  5. O raiz responde: "o servidor TLD do .com é X"
  6. O recursivo consulta o servidor TLD — quem tem google.com?
  7. O TLD responde: "o servidor autoritativo do Google é Y"
  8. O recursivo consulta o servidor autoritativo do Google
  9. Recebe: google.com → 142.250.78.14
  10. Salva no cache e repassa para você

Todo esse processo leva tipicamente entre 20–200ms na primeira consulta e <1ms nas seguintes (cache).

DHCP — Atribuição Automática de IP

📖 Explicação

DHCP (Dynamic Host Configuration Protocol) é o protocolo responsável por atribuir automaticamente configurações de rede para dispositivos que se conectam. Sem ele, você precisaria configurar manualmente em cada dispositivo: IP, máscara, gateway e DNS.

O processo DORA:

  1. D — Discover: o cliente envia broadcast "tem algum servidor DHCP aqui?"
  2. O — Offer: o servidor oferece um IP disponível
  3. R — Request: o cliente aceita a oferta
  4. A — Acknowledge: o servidor confirma e cede o IP por um lease time (tempo de aluguel)

O DHCP usa as portas UDP 67 (servidor) e 68 (cliente).

HTTP vs HTTPS

📖 Explicação

HTTP (Hypertext Transfer Protocol) é o protocolo da web. Define como clientes fazem requisições e servidores respondem. Opera na camada de Aplicação, porta 80.

HTTPS é o HTTP com uma camada TLS por baixo. O "S" significa Secure. Opera na porta 443. A diferença prática:

  • HTTP: dados em texto puro — qualquer intermediário pode ler. Se você fizer login via HTTP em um café, quem capturar o tráfego vê sua senha.
  • HTTPS: dados criptografados — intermediários veem apenas bytes embaralhados. Sem a chave de sessão, é inútil.

Métodos HTTP mais usados: GET (busca), POST (envia dados), PUT/PATCH (atualiza), DELETE (remove).

TLS — Transport Layer Security

📖 Explicação

O TLS (sucessor do SSL) oferece três garantias fundamentais:

  • Confidencialidade: os dados são criptografados. Sem a chave, não se lê.
  • Autenticação: o certificado digital prova que você está falando com quem pensa. Emitido por uma CA (Certificate Authority) confiável.
  • Integridade: um HMAC (hash) verifica que os dados não foram alterados no caminho.

No handshake TLS, cliente e servidor:

  1. Negociam a versão do protocolo e os algoritmos de criptografia
  2. O servidor envia seu certificado digital
  3. O cliente verifica a autenticidade do certificado com a CA
  4. Ambos derivam uma chave de sessão usando criptografia assimétrica (sem nunca transmitir a chave)
  5. A partir daqui, toda comunicação usa criptografia simétrica com a chave de sessão
⚡ TLS ≠ SSL

O SSL (Secure Sockets Layer) é o predecessor do TLS e está descontinuado por vulnerabilidades. Quando alguém diz "certificado SSL", na prática usa TLS 1.2 ou 1.3. TLS 1.3 (2018) é o padrão atual — mais rápido e seguro que versões anteriores.

SIP — Session Initiation Protocol

📖 Explicação

O SIP é o protocolo de sinalização usado em VoIP (Voice over IP) — a tecnologia por trás de ligações via internet (Skype, WhatsApp, Teams, sistemas PABX IP).

O SIP cuida de: iniciar a chamada, negociar codecs de áudio/vídeo, transferir chamadas e encerrar sessões. O áudio em si viaja em outro protocolo, o RTP (Real-Time Protocol).

Portas: 5060 (UDP/TCP não criptografado) e 5061 (TCP com TLS — SIP seguro).

Se você trabalha com telefonia, PABX virtual, ou call centers, vai encontrar SIP com frequência.

⚠️ SIP na internet sem TLS

Sistemas SIP sem criptografia (porta 5060 exposta) são alvos comuns de ataques de escuta e desvio de chamadas. Sempre use SIP sobre TLS (5061) e autenticação forte em sistemas de produção.
