Fundamentos de Redes

Antes de estudar protocolos e arquiteturas complexas, você precisa dominar os blocos básicos: o que é uma rede, como os dispositivos se identificam, e como os dados viajam.

O que é uma rede de computadores?

📖 Explicação

Uma rede de computadores é um conjunto de dois ou mais dispositivos — computadores, celulares, impressoras, roteadores — que estão interconectados e conseguem trocar dados entre si.

Essa conexão pode ser física (cabo Ethernet, fibra óptica) ou sem fio (Wi-Fi, 4G/5G). O que importa é que os dispositivos consigam se comunicar segundo um conjunto de regras chamadas protocolos.

A internet, por exemplo, não é uma rede única — é uma rede de redes: bilhões de dispositivos conectados através de milhões de redes menores que se interligam.

💡 Analogia — Sistema Postal

Pense na rede como um sistema postal global. Cada casa (dispositivo) tem um endereço único. Quando você quer mandar uma carta (dado), coloca no envelope (pacote), escreve o endereço do destinatário, e entrega nos Correios (roteador). Os Correios decidem o melhor caminho para que a carta chegue ao destino.

Sem endereços, os Correios não sabem para onde entregar. Sem o sistema de rotas, a carta nunca chega. Redes funcionam da mesma forma.

📝 Resumo
  • Rede = dispositivos interconectados que trocam dados
  • A conexão pode ser cabeada ou sem fio
  • Protocolos são as regras que permitem a comunicação
  • A internet é uma rede de redes

Cliente vs. Servidor

📖 Explicação

Na comunicação em rede, geralmente há dois papéis distintos:

  • Cliente: o dispositivo que inicia uma requisição. Ele pede algo — uma página web, um arquivo, um e-mail.
  • Servidor: o dispositivo que aguarda requisições e as processa, enviando uma resposta.

Importante: esses são papéis, não tipos de hardware. O mesmo computador pode ser cliente em uma conversa (quando acessa um site) e servidor em outra (quando outros acessam um serviço que ele hospeda).

💡 Analogia — Restaurante

Você (cliente) senta à mesa e chama o garçom. O garçom leva seu pedido para a cozinha (servidor). A cozinha processa o pedido e o garçom traz o prato de volta para você.

No modelo cliente-servidor: você é o navegador, o garçom é a rede, e a cozinha é o servidor web. O prato pronto é a página HTML que você recebe.

🔍 Exemplo Concreto

Quando você digita google.com no navegador:

  • Seu navegador (cliente) envia uma requisição HTTP GET para o servidor do Google
  • O servidor do Google processa a requisição
  • O servidor responde com o HTML, CSS e JavaScript da página
  • Seu navegador renderiza (desenha) a página na tela

Todo esse processo acontece em menos de 1 segundo na maioria das conexões modernas.

⚡ O que você precisa guardar

Cliente inicia, servidor responde. O cliente sempre dá o primeiro passo. O servidor fica esperando (escutando numa porta) até alguém pedir algo.

Endereço IP

📖 Explicação

IP significa Internet Protocol. O endereço IP é um número único que identifica cada dispositivo em uma rede — sem ele, ninguém sabe para onde enviar os dados.

Existem duas versões em uso hoje:

  • IPv4: formato 192.168.1.10 — 4 grupos de números de 0 a 255, separados por ponto. Tecnicamente são 32 bits de informação. Suporta cerca de 4,3 bilhões de endereços (que já estão esgotados).
  • IPv6: formato 2001:0db8:85a3::8a2e:0370:7334 — 128 bits. Suporta 340 undecilhões de endereços (praticamente infinito).
💡 Analogia — Endereço de casa

O IP é como o endereço de uma casa: "Rua das Flores, 42, São Paulo". Sem esse endereço, o carteiro não sabe onde entregar a correspondência. Com o endereço, o sistema postal (a rede) consegue rotear a carta até o destino correto.

🔍 Exemplo Concreto

Para ver seu IP local, abra o terminal e digite:

# Windows
ipconfig

# Linux / Mac
ip addr
# ou
ifconfig

Você vai ver algo como 192.168.1.105. Esse é seu endereço dentro da rede local. Para ver seu IP público (como a internet te enxerga), acesse whatismyip.com.

⚠️ Erros comuns de iniciantes

Confundir IP local com IP público: seu roteador tem um IP público (ex: 201.34.56.78) que é visível na internet. Seus dispositivos em casa têm IPs privados (ex: 192.168.1.x) que não saem diretamente para a internet. Isso é feito pelo NAT (veremos no Módulo 3).

Achar que IP é fixo: na maioria dos casos, o IP muda. Seu provedor pode trocar seu IP público, e o DHCP pode dar um IP diferente ao seu notebook a cada conexão.

Portas de Rede

📖 Explicação

O endereço IP identifica o computador na rede. Mas um computador roda dezenas de serviços ao mesmo tempo — navegador web, cliente de e-mail, servidor FTP. Como os dados chegam ao serviço certo?

É aí que entram as portas. Uma porta é um número de 0 a 65535 que identifica qual serviço/aplicação dentro do computador deve receber aquele pacote.

Endereços famosos: HTTP usa porta 80, HTTPS usa 443, SSH usa 22, DNS usa 53.

💡 Analogia — Apartamento

O endereço IP é o endereço do prédio ("Av. Paulista, 1000"). A porta é o número do apartamento ("Apto 42"). A carta (pacote) chega no prédio pelo IP, mas o porteiro (sistema operacional) olha para o número do apartamento (porta) para saber a qual morador entregar.

⚡ O que você precisa guardar
  • IP: identifica o computador (endereço do prédio)
  • Porta: identifica o serviço (número do apartamento)
  • Um endereço completo na rede é: IP:porta — ex: 192.168.1.5:80
  • Portas de 0 a 1023 são "well-known" e reservadas para serviços padrão

Pacotes de Dados

📖 Explicação

Dados na rede não viajam em um único bloco — eles são divididos em pedaços menores chamados pacotes. Cada pacote carrega:

  • Cabeçalho (Header): metadados — endereço de origem, endereço de destino, número de sequência, protocolo, checksum (para verificação de erros).
  • Payload (dados): o conteúdo em si — um pedaço do arquivo, parte da página web, etc.

Os pacotes podem seguir caminhos diferentes pela rede e chegar fora de ordem. O protocolo TCP (que veremos no Módulo 4) garante que eles sejam remontados na ordem correta no destino.

💡 Analogia — Carta em pedaços

Imagine que você precisa enviar uma carta enorme para um amigo que mora do outro lado do país, mas só tem envelopes pequenos. Você divide a carta em 10 partes, numera cada parte (1/10, 2/10, 3/10...) e coloca em envelopes separados. Cada envelope pode ir por uma rota diferente, mas seu amigo consegue remontar a carta na ordem certa quando todos chegam.

🔍 Exemplo Concreto

Quando você baixa um vídeo de 100 MB:

  • O arquivo é dividido em milhares de pacotes de ~1500 bytes cada
  • Cada pacote tem o endereço IP de origem (servidor), destino (seu computador) e seu número de sequência
  • Os pacotes viajam pela rede, possivelmente por caminhos diferentes
  • Seu computador recebe os pacotes, verifica a integridade e remonta o arquivo na ordem

Latência, Largura de Banda e Jitter

📖 Explicação

Latência (RTT — Round-Trip Time): é o tempo que um pacote leva para ir do ponto A ao ponto B e voltar. Medida em milissegundos (ms). Quanto menor, melhor.

  • < 20ms: excelente (fibra local)
  • 20–80ms: bom (DSL, cabo)
  • 80–200ms: perceptível (pode afetar jogos online)
  • > 200ms: ruim (VoIP com atraso notável)

Largura de Banda (Bandwidth): é a capacidade máxima de transferência de dados de um link, medida em Mbps ou Gbps. É o tamanho do "cano" — quanto maior, mais dados passam simultaneamente. Não confunda com velocidade!

Jitter: é a variação na latência entre pacotes consecutivos. Se o primeiro pacote demora 20ms, o segundo 25ms e o terceiro 15ms, o jitter é essa oscilação. Alto jitter é devastador para VoIP e video chamadas.

💡 Analogia — Estrada

Latência é o tempo de viagem de São Paulo ao Rio de Janeiro (distância e velocidade da estrada). Largura de banda é o número de faixas da rodovia — mais faixas, mais carros (dados) ao mesmo tempo. Jitter é quando o trânsito para e anda, para e anda — os carros chegam em intervalos irregulares.

🔍 Exemplo Concreto

Execute no terminal:

ping -c 5 8.8.8.8

A saída mostra o RTT de cada pacote (ex: time=12.4 ms). O min/avg/max/mdev no final mostra o mínimo, médio, máximo e o desvio padrão — esse último é uma medida de jitter.

⚠️ Erros comuns de iniciantes

"Minha internet é 100Mbps, por que o download está lento?" — Largura de banda alta não garante latência baixa. Se o servidor está sobrecarregado ou sua distância ao servidor é grande, o download pode ser lento mesmo com banda boa. São problemas diferentes.

Confundir Mbps com MB/s: 100 Mbps (megabits por segundo) = 12,5 MB/s (megabytes por segundo). Provedores anunciam em bits, gerenciadores de download mostram em bytes. Divida por 8 para converter.

⚡ O que você precisa guardar
  • Latência: tempo de viagem do pacote (ms) — afeta experiência em tempo real
  • Banda: capacidade do link (Mbps/Gbps) — afeta throughput (volume)
  • Jitter: variação da latência — killer de VoIP e jogos online
  • Pode ter banda alta e latência alta ao mesmo tempo — são características independentes
