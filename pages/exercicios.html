Exercícios Práticos

Aqui a meta é pensar como alguém que diagnostica rede no mundo real. Cada exercício traz um cenário, comandos úteis para Windows e Linux e uma solução comentada para você comparar seu raciocínio.

📖 Como estudar estes exercícios

Leia o cenário, tente formular sua hipótese, só depois expanda a solução. O valor aqui não está em decorar comandos isolados, e sim em aprender a ordem mental de investigação.

  • Se possível, rode os comandos em uma máquina real
  • Compare saídas e pense no que cada comando confirma ou elimina
  • Se errar a hipótese, ótimo: isso mostra onde seu modelo mental ainda está incompleto
1. Ping para IP funciona, mas domínio não abre
WindowsLinux

Cenário: o usuário consegue ping 8.8.8.8, mas ping google.com falha e o navegador não abre sites por nome.

Comandos úteis
nslookup google.com
ipconfig /all
cat /etc/resolv.conf
ping 8.8.8.8
ping google.com
O problema mais provável é DNS, não conectividade IP. O host alcança IPs diretamente, então a pilha TCP/IP e a rota básica estão funcionando. O próximo passo é checar qual servidor DNS está configurado, se responde e se retorna IPs válidos. Se necessário, teste temporariamente com 8.8.8.8 ou 1.1.1.1.
2. Site não carrega, mas outros funcionam
WindowsLinux

Cenário: o usuário abre vários sites normalmente, mas um domínio específico sempre falha.

Comandos úteis
nslookup api.empresa.com
curl -I https://api.empresa.com
tracert api.empresa.com
traceroute api.empresa.com
Esse cenário costuma apontar para problema localizado no serviço, no DNS daquele domínio ou em bloqueio específico. Se o domínio resolve mas curl retorna timeout ou connection refused, o DNS pode estar correto e o problema passa a ser porta, firewall, serviço fora do ar ou balanceador mal configurado.
3. Conexão local funciona, internet não
WindowsLinux

Cenário: o notebook pinga o gateway normalmente, mas não alcança nenhum IP externo.

Comandos úteis
ipconfig
route print
ip route
ping 192.168.1.1
ping 8.8.8.8
A rede local está viva, então a suspeita principal recai sobre gateway, rota padrão, NAT do roteador ou link WAN do provedor. Se o cliente chega no gateway, mas nada sai para fora, o defeito provavelmente está além da máquina do usuário.
4. Porta do serviço parece fechada
WindowsLinux

Cenário: um sistema web interno deveria responder em 10.0.0.20:443, mas o acesso falha.

Comandos úteis
curl -I https://10.0.0.20
netstat -ano
ss -tulpn
telnet 10.0.0.20 443
Aqui a pergunta não é só "tem rede?", e sim existe processo ouvindo na porta certa? Se a conexão é recusada, o host existe mas o serviço não está escutando ou um firewall local está devolvendo reset. Se há timeout, pense em ACL, firewall de rede ou rota quebrada.
5. Latência alta só depois de alguns hops
Análise

Cenário: o ping local é excelente, mas usuários reclamam de lentidão em um serviço hospedado fora da empresa.

Comandos úteis
ping 8.8.8.8
traceroute servico.externo.com
tracert servico.externo.com
Se os primeiros saltos estão rápidos e a latência explode mais à frente, a causa provável está em trânsito externo, link de longa distância ou congestionamento fora da LAN. Isso ajuda a evitar o erro clássico de culpar Wi-Fi ou switch local sem evidência.
6. Suspeita de DNS interno errado em ambiente corporativo
WindowsLinux

Cenário: o sistema interno abre em algumas máquinas, mas em outras resolve para IP diferente ou desatualizado.

Comandos úteis
nslookup intranet.empresa.local
nslookup intranet.empresa.local 8.8.8.8
ipconfig /flushdns
resolvectl status
Pense em cache DNS local, DNS interno autoritativo, split-horizon DNS ou registro desatualizado. Quando máquinas diferentes recebem respostas diferentes, o problema costuma estar menos no navegador e mais na cadeia de resolução.
